工具入口

先體驗「AI 帶你拆步驟＋一起做」的感覺

💡 Antigravity（入門體驗）

我會先推薦 Antigravity，原因很簡單：它比較像有人幫你「拆步驟＋帶你做」，而不是只丟一段答案。

展開了解如何使用

🎯 如何開始：

  1. 前往 antigravity.anthropic.com
  2. 登入後，直接在對話框輸入你的想法
  3. 使用下方的 prompt 範例，讓 AI 幫你拆解步驟

📝 我最常用它做三件事：

  • 把一個想法拆成可執行的小步驟
  • 讓 AI 先把「我接下來要做什麼」講清楚
  • 讓 AI 幫我把需求變成清楚的文件草稿
💬 第一次使用可以這樣問：
我想做一個小功能/小作品：<一句話描述>
我是新手/我想降低風險。請先給我 5–10 個小步驟，每步只做一件事。
請只帶我做第 1 步，並告訴我完成後應該看到什麼結果。

小技巧：每次只讓它做一步，確認跑得起來再往下，這樣出錯時才好追。

🚀 Claude Code（日常主力）

我最常用 Claude Code（整合在 VS Code），因為它更像「在專案裡幫你一起改」：能沿著 repo 結構走，比較適合做重構、補測試、修 bug、補文件。

展開了解如何使用

🎯 如何開始：

  1. 在 VS Code 安裝「Claude Code」擴充套件
  2. 登入後，用 Cmd/Ctrl + Shift + P 開啟命令面板
  3. 選擇「Claude Code: Open Chat」開始對話
  4. 在專案資料夾中使用，它會自動讀取你的檔案結構

💡 我最常的搭配方式：

  • 先用 ChatGPT/Antigravity：拆解任務、寫清楚文件
  • 再用 Claude Code：在 IDE 裡一步步實作到可跑可驗證
💬 實作時可以這樣問：
先不要一次改很多。請先列出你的修改計畫（要改哪些檔案、每步做什麼）。
我確認後你再從第 1 步開始，且每一步都要能跑得起來。
如果出錯，我會貼錯誤訊息給你，你再針對錯誤修。

小技巧：在對話中可以用 @檔案名稱 讓它看特定檔案，用 @folder 讓它看整個資料夾。

核心技巧

三個立刻有效的技巧

📋 技巧 1: 先要步驟清單再動手

我用 AI 最大的心得是：先把需求寫成一頁文件，AI 產出會穩非常多。

我想做：<一句話>
請幫我寫一頁「最小需求文件」，包含 Goal / User flow / Scope（含不做清單）/ Acceptance（可驗收條件）。
請用白話，避免術語。
為什麼這樣做？深入理解文件先行

因為 AI 最容易翻車的情況是：需求在腦袋、聊天、零碎訊息裡。它只能猜，然後就會「看起來合理但其實不對」。

📝 最小需求文件（入門版）：

  • Goal：我要做什麼（1 句話）
  • User flow：使用者怎麼操作（3–5 行）
  • Scope：這次做哪些、不做哪些
  • Acceptance：怎樣算完成（可驗收條件）

📚 進階文件（複雜專案）：

當專案變大，你可能需要更詳細的文件結構：

  • Technical Context：使用哪些技術、為什麼選它們
  • Architecture：系統如何拆分、模組如何互動
  • Data Model：資料結構、關聯、限制
  • Edge Cases：邊界情況、錯誤處理、效能考量
  • Testing Strategy：怎麼驗證、測試案例清單

💡 Context Engineering 概念：

「給對的資訊」比「給很多資訊」重要。不是把整個專案都丟給 AI，而是：

  • 分層給：先給高層架構 → 再給相關模組 → 最後給具體實作
  • 指出重點：明確說「這個檔案是 X 的入口」、「這段程式處理 Y 邏輯」
  • 限制範圍：「這次只改認證流程，不要動到其他部分」

⚠️ 常見錯誤：

  • 文件太簡略：只寫「做一個登入功能」→ AI 會自己腦補很多細節，結果跟你想的不一樣
  • 文件太複雜：一次寫 10 頁規格 → AI 反而抓不到重點，或是做了一堆你不需要的功能
  • 沒有 Scope：沒說「不做什麼」→ AI 可能順手加了很多「看起來該做」的功能

✨ 實際案例：做一個待辦事項 App

Goal：一個簡單的待辦清單，能新增、刪除、標記完成

User flow：

  1. 打開頁面 → 看到輸入框和已有的待辦
  2. 輸入文字 → 按 Enter → 新增到清單
  3. 點擊待辦 → 標記完成/取消完成
  4. 點 X → 刪除待辦

Scope：只做單頁面，不做登入、不做資料庫、不做分類

Acceptance：能新增、刪除、標記完成，重新整理後資料還在（用 localStorage）

記住：文件不是一次寫完，而是在過程中逐步補充。從最小版本開始，遇到問題再回來補。

✅ 技巧 2: 每做一小段就驗證

不要一次改很多。先要計畫，再要最小修改，立刻跑起來驗證。

請先給計畫（3–6 步），再給第 1 步的最小 patch。
我會跑起來驗證並回報結果，你再給下一步。
完整流程：Plan → Patch → Run

這是我最穩的改 code 節奏：

  1. Plan（計畫）：先要一個 3–6 步的計畫，了解全貌
  2. Patch（最小修改）：只要第一步的修改，不要一次全部
  3. Run（驗證）：立刻跑起來看結果 【註】跑起來＝build/test/打開畫面看結果
  4. 回報：告訴 AI「成功」或「錯誤訊息」
  5. 重複：AI 再給下一步，繼續循環

⚠️ 反例：一次改太多會發生什麼？

壞習慣：「請幫我完成整個登入功能」

→ AI 一次給你改了 5 個檔案、新增 3 個功能

→ 跑起來出錯了，你不知道是哪一步出問題

→ 只能全部回頭重來，浪費時間

好習慣：「請先給計畫，再給第一步」

→ AI 給計畫：1) 建 form 2) 加驗證 3) 串 API 4) 處理錯誤

→ 第一步：只改一個檔案，加一個 form

→ 跑起來，畫面出現了 ✓ → 繼續下一步

→ 出錯了也好追，只要檢查最後一步的改動

💡 如何判斷「最小」？

  • 只改一個功能：不要同時加新功能和改舊功能
  • 能獨立驗證：這一步做完應該能看到某個明確的改變（畫面、log、測試通過）
  • 改動少於 3 個檔案：如果需要改很多檔案，可能這步太大了，再拆小一點

記住：小步快跑不是慢，而是穩。出錯時能快速定位問題，反而比一次改完快很多。

🔍 技巧 3: 卡住時先要可能原因

我不會直接問「怎麼修」，而是問：可能原因有哪些（按機率排序）+ 每個原因最簡單怎麼驗證。

我遇到錯誤：<貼錯誤訊息>
請列 3–5 個可能原因（按機率排序），每個原因給最小驗證步驟。
等我回報驗證結果後，你再給最推薦修法。
為什麼要「原因排序」而不是直接修？

因為直接問「怎麼修」，AI 只能猜最有可能的原因，如果猜錯你就浪費時間了。但如果先列出所有可能原因，你可以快速驗證找出真兇，再對症下藥。

✨ 實際案例：畫面顯示錯誤

錯誤訊息：Cannot read property 'name' of undefined

AI 列出可能原因（按機率）：

  1. API 還沒回傳資料（機率 60%） → 驗證：加 console.log 看資料何時載入
  2. API 回傳的資料結構不對（機率 25%） → 驗證：console.log 看實際資料長什麼樣
  3. 變數拼錯字（機率 10%） → 驗證：檢查變數名稱
  4. 條件判斷有問題（機率 5%） → 驗證：檢查 if 條件

結果：驗證第 1 項，發現確實是資料還沒載入就渲染了 → 加個 loading 狀態解決

💡 常見錯誤原因清單（備查）：

  • 資料相關：API 還沒回傳、資料結構不對、變數是 null/undefined
  • 時機相關：操作順序不對、非同步沒等完、DOM 還沒載入
  • 路徑相關：檔案路徑錯誤、import 路徑錯誤、URL 不對
  • 邏輯相關：條件判斷有誤、迴圈邊界錯誤、狀態沒更新
  • 環境相關：缺套件、版本不對、環境變數沒設定

⚠️ 除錯技巧：

  • 複製完整錯誤訊息：包含檔案路徑、行號、堆疊追蹤（stack trace）
  • 說明操作步驟：「我點了 X 按鈕 → 出現錯誤」而不是只貼錯誤
  • 一次只驗證一個原因：不要同時改多個地方，否則不知道哪個有效
  • 驗證完回報結果：「試了第 1 項，發現資料確實是空的」

記住：除錯不是猜謎，而是科學方法 — 假設 → 驗證 → 結論。原因排序就是幫你快速找到正確的假設。

進階工作方式

讓 AI 從聊天變成可重複的工具

🎯 Skill / Rule / MCP

這三個概念，是我把 AI 從「一次性聊天」升級成「可重複流程」的關鍵。

展開了解更多

Skill（技能）＝可重複的工作套路

把常見任務做成 skill，例如：產出需求文件、產出測試計畫、產出 code review checklist。

Rule（規則）＝限制 AI，降低亂改風險

我常用的 rule：

  • 不要重寫整個檔案，只改需要的部分
  • 先給計畫，再執行
  • 不確定就先問 2–3 個關鍵問題

MCP（工具連接）＝讓 AI 真的能「做事」

讓 AI 透過標準介面連到工具（repo、文件、資料庫），不是只能嘴砲。

常見問題

Q: 我看不懂程式在做什麼怎麼辦？ +

別急著看懂每一行，先試著「驗證結果」。

我自己的做法是：

  1. 先讓 AI 給我「這段 code 在做什麼」（1 句話）
  2. 再跑起來看結果是否符合預期
  3. 如果不懂某一段，再拿那段單獨問
這段程式的目的是什麼？（請用 1 句話白話說明）
如果我要測這段有沒有正常工作，最簡單的驗證方式是什麼？
Q: 出現錯誤訊息要怎麼問 AI？ +

不要只貼錯誤訊息，要加上「我做了什麼」。

完整的問法包含三部分：

  1. 我做了什麼操作
  2. 出現什麼錯誤（完整錯誤訊息）
  3. 我預期應該出現什麼結果
我在執行 [操作步驟]
出現錯誤：[完整錯誤訊息]
我預期應該 [預期結果]
請列出可能原因（按機率排序）和驗證方式。
Q: 我做到一半很亂怎麼整理？ +

先保存（git commit），再請 AI 幫你整理。

我的整理流程：

  1. 先 commit 目前的狀態（至少能回去）
  2. 列出「哪些檔案/功能是有用的」
  3. 請 AI 給出最小整理計畫（刪什麼、合併什麼）
  4. 一步步執行並驗證
我的專案現在有這些檔案：[列出檔案清單]
我想保留的功能是：[列出核心功能]
請給我最小的整理計畫（哪些可以刪、哪些可以合併）。
Q: AI 很貴怎麼辦？ +

用對方式，可以省很多 token。

我的省錢技巧：

  • 先寫文件（下次不用重複解釋）
  • 不要每次都貼整個專案，只貼相關部分
  • 用免費工具做前期規劃（Antigravity），再用付費工具精修
  • 把常用流程做成 skill（可重複使用）
我不會貼整個專案。你先告訴我你需要哪 3 段資訊（檔案路徑/函式/錯誤 log）。
我貼給你後，你再給最小 patch。
Q: 我不知道怎麼驗證對不對？ +

讓 AI 告訴你「完成後應該看到什麼」。

驗證三步驟：

  1. 問 AI：「完成這步後，我應該看到什麼結果？」
  2. 實際執行並記錄結果
  3. 回報給 AI：「我看到 [實際結果]，是否正確？」
完成 [這個步驟] 後，我應該看到什麼結果？
請給我最簡單的驗證方式（不要太技術性）。
Q: AI 給的答案不能用怎麼辦？ +

先確認是「不能用」還是「不知道怎麼用」。

常見情況和對策：

  • 情況 1：跑起來出錯 → 貼完整錯誤訊息，用技巧 3 找原因
  • 情況 2：能跑但不符合需求 → 檢查是不是需求沒說清楚，回去補文件
  • 情況 3：看不懂 AI 的解法 → 問「請用白話解釋這段在做什麼」
  • 情況 4：方案太複雜 → 明確要求「請給最簡單的做法，不要過度設計」
你給的這段 [code/方案]，我 [遇到什麼問題]。
能不能換一個更簡單/更符合 [需求] 的做法？

記住：AI 不是萬能，但它可以無限次重試。不怕問題多，就怕不問。

Q: 如何判斷 AI 給的方案好不好？ +

用這四個標準快速判斷：

  1. 能不能跑起來？（最基本）
  2. 有沒有達成需求？（對照你的驗收條件）
  3. 我看得懂嗎？（太複雜的方案通常不是好方案）
  4. 容易改嗎？（如果之後要加功能，會不會很麻煩）

紅旗警訊（可能有問題的方案）：

  • 一次改了超過 5 個檔案
  • 引入了很多新的套件或工具
  • 解釋不清楚為什麼要這樣做
  • 沒有考慮錯誤處理
請用白話解釋你這個方案的優缺點。
有沒有更簡單的做法？為什麼選這個方案而不是簡單的？
Q: 專案變大變複雜怎麼辦？ +

專案大不可怕，可怕的是一開始沒有規劃。

管理大專案的技巧：

  • 模組化：把功能拆成獨立模組，每個模組可以單獨測試
  • 寫 README：說明專案結構、每個資料夾放什麼、如何開始
  • 版本控制：常常 commit，每個 commit 做一件事，寫清楚做了什麼
  • 定期整理：刪掉不用的檔案、合併重複的程式碼
我的專案現在有 [列出主要功能]。
請幫我規劃一個清楚的資料夾結構，讓我知道每個檔案該放哪裡。

記住：大專案是從小專案長出來的，一開始就保持整潔，後面就不用重構。

💡 順帶一提

這個教學網站本身也是用 AI 快速做出來的

我用 AI 開發最大的收穫不是「寫得更快」，而是：把做事方式變成可重複的流程（文件先行、分步驗證、規則化），再用實際跑起來把風險壓下來。

展開看這個網站的開發流程

🔨 實際開發流程：

  1. 需求整理：先把 2 份文件（需求規格 + 內容文稿）整合成清楚的開發規格
  2. 規劃分解：讓 AI 把任務拆成 4 個階段（基礎建設 → 結構建立 → 內容填充 → 部署上線）
  3. 分階段執行：每個階段做完就測試，確認可以跑才往下
  4. 品質檢查：完成後讓 AI 重新檢查是否符合驗收標準
  5. 部署上線：推送到 GitHub Pages，確認線上版本正常

💡 關鍵心法：

  • 文件先行：不急著寫 code，先把「要做什麼」說清楚
  • 小步快跑：每次改一點點，立刻跑起來驗證
  • 問對問題：不是問「幫我做」，而是問「給我計畫 → 我確認 → 執行第一步」

整個網站從零到上線大約 2 小時，但重點不是快，而是過程中每一步都可控、可驗證。
