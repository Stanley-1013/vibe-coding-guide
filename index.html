核心技巧

三個立刻有效的技巧

📋 技巧 1: 先要步驟清單再動手

我用 AI 最大的心得是：先把需求寫成一頁文件，AI 產出會穩非常多。

我想做：<一句話>
請幫我寫一頁「最小需求文件」，包含 Goal / User flow / Scope（含不做清單）/ Acceptance（可驗收條件）。
請用白話，避免術語。
為什麼這樣做？

因為 AI 最容易翻車的情況是：需求在腦袋、聊天、零碎訊息裡。它只能猜，然後就會「看起來合理但其實不對」。

最小需求文件通常只要包含：

  • Goal：我要做什麼（1 句話）
  • User flow：使用者怎麼操作（3–5 行）
  • Scope：這次做哪些、不做哪些
  • Acceptance：怎樣算完成（可驗收條件）

✅ 技巧 2: 每做一小段就驗證

不要一次改很多。先要計畫，再要最小修改，立刻跑起來驗證。

請先給計畫（3–6 步），再給第 1 步的最小 patch。
我會跑起來驗證並回報結果，你再給下一步。
完整流程：Plan → Patch → Run

這是我最穩的改 code 節奏：

  1. 先要計畫（Plan）
  2. 再要最小修改（Patch）
  3. 立刻跑起來驗證（Run） 【註】跑起來＝build/test/打開畫面看結果

🔍 技巧 3: 卡住時先要可能原因

我不會直接問「怎麼修」，而是問：可能原因有哪些（按機率排序）+ 每個原因最簡單怎麼驗證。

我遇到錯誤：<貼錯誤訊息>
請列 3–5 個可能原因（按機率排序），每個原因給最小驗證步驟。
等我回報驗證結果後，你再給最推薦修法。

進階工作方式

讓 AI 從聊天變成可重複的工具

🎯 Skill / Rule / MCP

這三個概念，是我把 AI 從「一次性聊天」升級成「可重複流程」的關鍵。

展開了解更多

Skill（技能）＝可重複的工作套路

把常見任務做成 skill，例如：產出需求文件、產出測試計畫、產出 code review checklist。

Rule（規則）＝限制 AI，降低亂改風險

我常用的 rule：

  • 不要重寫整個檔案，只改需要的部分
  • 先給計畫，再執行
  • 不確定就先問 2–3 個關鍵問題

MCP（工具連接）＝讓 AI 真的能「做事」

讓 AI 透過標準介面連到工具（repo、文件、資料庫），不是只能嘴砲。

常見問題

Q: 我看不懂程式在做什麼怎麼辦？ +

別急著看懂每一行，先試著「驗證結果」。

我自己的做法是：

  1. 先讓 AI 給我「這段 code 在做什麼」（1 句話）
  2. 再跑起來看結果是否符合預期
  3. 如果不懂某一段，再拿那段單獨問
這段程式的目的是什麼？（請用 1 句話白話說明）
如果我要測這段有沒有正常工作，最簡單的驗證方式是什麼？
Q: 出現錯誤訊息要怎麼問 AI？ +

不要只貼錯誤訊息，要加上「我做了什麼」。

完整的問法包含三部分：

  1. 我做了什麼操作
  2. 出現什麼錯誤（完整錯誤訊息）
  3. 我預期應該出現什麼結果
我在執行 [操作步驟]
出現錯誤：[完整錯誤訊息]
我預期應該 [預期結果]
請列出可能原因（按機率排序）和驗證方式。
Q: 我做到一半很亂怎麼整理？ +

先保存（git commit），再請 AI 幫你整理。

我的整理流程：

  1. 先 commit 目前的狀態（至少能回去）
  2. 列出「哪些檔案/功能是有用的」
  3. 請 AI 給出最小整理計畫（刪什麼、合併什麼）
  4. 一步步執行並驗證
我的專案現在有這些檔案：[列出檔案清單]
我想保留的功能是：[列出核心功能]
請給我最小的整理計畫（哪些可以刪、哪些可以合併）。
Q: AI 很貴怎麼辦？ +

用對方式，可以省很多 token。

我的省錢技巧：

  • 先寫文件（下次不用重複解釋）
  • 不要每次都貼整個專案，只貼相關部分
  • 用免費工具做前期規劃（Antigravity），再用付費工具精修
  • 把常用流程做成 skill（可重複使用）
我不會貼整個專案。你先告訴我你需要哪 3 段資訊（檔案路徑/函式/錯誤 log）。
我貼給你後，你再給最小 patch。
Q: 我不知道怎麼驗證對不對？ +

讓 AI 告訴你「完成後應該看到什麼」。

驗證三步驟：

  1. 問 AI：「完成這步後，我應該看到什麼結果？」
  2. 實際執行並記錄結果
  3. 回報給 AI：「我看到 [實際結果]，是否正確？」
完成 [這個步驟] 後，我應該看到什麼結果？
請給我最簡單的驗證方式（不要太技術性）。

💡 順帶一提

這個教學網站本身也是用 AI 快速做出來的

我用 AI 開發最大的收穫不是「寫得更快」，而是：把做事方式變成可重複的流程（文件先行、技能化、規則化、工具化），再用跑起來驗證把風險壓下來。

展開看這個網站的開發流程

實際流程：

  1. 需求文件：整合 2 份文件（需求規格 + 內容文稿）
  2. PFC 規劃：使用 han-agents 多代理系統分解任務
  3. 分波執行：Wave 1 基礎 → Wave 2 結構 → Wave 3 內容 → Wave 4 部署
  4. QA 驗證：Critic agent 檢查驗收標準

整個過程展示了：先聊清楚需求 → 產出文件 → 分步實作 → 每步驗證。
